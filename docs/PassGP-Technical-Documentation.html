1 Introduction & purpose

PassGP AI Oral Examiner is a web platform that lets medical candidates sit a realistic, spoken oral examination with an AI examiner. The candidate talks to the examiner in real time (voice), is marked against each exam's own marking criteria sourced from Jotform, and receives an elegantly formatted PDF report by email. Administrators (PassGP) curate the clinical case bank, define each exam's personality and marking standard, and review analytics — all from a dedicated admin console, with no code changes required for day-to-day operation.

Design principles

  • Secrets stay server-side. The browser apps never hold API keys; the backend mints short-lived tokens.
  • Marking follows the source. Scores come from each case's Jotform criteria (real totals), never a generic /10.
  • Configurable without code. Examiner behaviour, mark schemes and pass standards are edited in the admin.
  • Single source of truth. One Postgres database (Supabase) backs all three apps.

Request lifecycle of a live exam

  • 1. Token mint. The chatbot calls POST /api/realtime/session. The backend builds the examiner prompt (case + personality + standard), asks OpenAI for a short-lived ephemeral key, and returns it. The real OPENAI_API_KEY never leaves the server.
  • 2. Direct voice. The browser opens a WebRTC connection straight to OpenAI using the ephemeral key — low latency, no audio through our server.
  • 3. Transcript. Speech-to-text events stream back over the data channel; the chatbot collects the transcript.
  • 4. Grading. On finish, the chatbot calls POST /api/feedback. The backend grades the transcript against the case's marking criteria (OpenAI Chat) and saves the session to Supabase.
  • 5. Report + email. The report page renders charts, captures the report to PDF, and calls POST /api/email-report to email it.

5 End-to-end methodology (the exam lifecycle)

#StageWhat happens
1RegistrationCandidate enters name + email + chooses an exam (e.g. StAMPS, RACGP CCE). Exams shown come from GET /api/exam-profiles (only exams that have cases).
2Case selectionThe examiner greets, states how many cases are available, and asks "which case number, or 'any'?". A specific case can also be forced via the embed URL (?caseId= / ?formId=).
3Live oral examTwo-way voice with the AI examiner. It plays examiner and patient (per the exam's role mode), probes to the exam's standard, and supports barge-in, a countdown timer and auto-termination.
4MarkingThe transcript is graded against the case's marking criteria and real total (e.g. 17), with the exam's mark scheme + standard deciding pass/fail. Killer/unsafe items force an automatic fail.
5ReportA visual report (gauge, competency radar, domain bars, strengths, missed items) is shown and saved.
6EmailThe exact on-screen report (with charts) is emailed to the candidate as a PDF.
7HistoryCandidates can look up their past attempts by email; admins see every session + analytics.

Mock circuits

Launching with ?mock=N runs an N-station circuit back-to-back; each station is graded and saved separately, with a combined circuit summary at the end.

6 The AI examiner & marking engine

The examiner prompt is assembled at session time from: the chosen exam's profile (personality, role mode, mark scheme, standard), the selected case (scenario, patient script, questions, criteria), and global AI config (voice, difficulty, any admin override).

How marking reads the Jotform

  • SAQ / OSCE cases: every point under "Correct possible answers" = 1 mark; the total is the number of points (e.g. 17), not 10.
  • MCQ cases: the (Score: N) annotations are summed.
  • Pass mark: defaults to 50% of the case total; overridable per case, and the exam's standard guides borderline pass/fail.
  • Exam sessions: because the candidate picks a case verbally, the grader is given the exam's cases (deterministic set), identifies which one the transcript covers, and marks against that case's scheme.

11 Security model

  • Secret isolation. API keys exist only on the backend. The browser receives short-lived ephemeral tokens for voice.
  • CORS allow-list. Only the PassGP apps (and configured domains) may call the API.
  • Row-Level Security. Direct writes to the case bank from the browser are revoked; all create/edit/delete go through authenticated backend endpoints (service role).
  • Admin auth. Supabase Auth; every admin endpoint checks the JWT and admin_users membership; superadmin role for sensitive actions.
  • Password reset. Secure email-link flow (no password set from email alone).
  • Analytics API. Separate shared key, never the database key.

12 Deployment & operations

  • CI/CD. GitHub main → Vercel auto-deploys all three projects.
  • Migrations. SQL files in server/db/migrations/, applied with node db/runSql.mjs <file> via the Supabase pooler.
  • Re-import. Admin → Questions → "Import from Jotform" → Import all re-reads every case's marking scheme.
  • Admin login. admin@passgp.au (reset via the login page → emailed link).
Change behaviour without code: exam personalities / mark schemes / pass standards → Exam Profiles; voice, models, role mode → AI Configuration; which cases are live → Training Panel; keys & email → Settings.
